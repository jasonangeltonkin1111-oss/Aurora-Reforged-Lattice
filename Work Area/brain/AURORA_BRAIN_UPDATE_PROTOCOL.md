# AURORA BRAIN UPDATE PROTOCOL

Run: ARL-RUN009G  
Status: Mandatory for serious ARL runs

---

## Purpose

This protocol defines when and how the Brain Guidebooks are updated.

---

## Brain Update Question

Every serious run must ask:

```text
Did this run discover a durable lesson that should improve future work?
```

If yes, update the correct Brain Guidebook.

If no, say no brain update was needed in the report.

---

## Promotion Criteria

A lesson may enter the brain only if it does at least one:

```text
prevents known failure
improves source safety
improves compile/runtime proof discipline
improves prompt quality
improves research quality
improves trader-output quality
clarifies source ownership
adds a falsifier or acceptance test
protects version/report/Git hygiene
```

---

## Rejection Criteria

Do not add if it is:

```text
run-specific noise
generic advice
duplicate content
emotional commentary
unverified speculation
ornamental doctrine
already captured elsewhere
```

---

## Update Method

Use this order:

1. Identify durable lesson.
2. Choose guidebook.
3. Check whether lesson already exists.
4. Update existing section where possible.
5. Add a concise entry to `AURORA_RUN_LESSONS_LEDGER.md`.
6. Mention in run report.

---

## Contradiction Rule

If a new lesson conflicts with an existing brain rule:

```text
do not add a second conflicting rule
log contradiction
decide which rule survives
update the guidebook directly
record why
```

---

## Brain vs Report

Reports contain the full story of a run.

Brain contains distilled reusable laws.

Do not paste reports into brain.

---

## Brain vs Office

Office owns formal project records.

Brain owns durable operating intelligence.

Do not use brain as a substitute for office ledgers.
