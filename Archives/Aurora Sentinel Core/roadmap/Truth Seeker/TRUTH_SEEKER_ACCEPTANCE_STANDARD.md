# Truth Seeker Acceptance Standard

**Updated:** 2026-04-27  
**Purpose:** Defines when a future phase/run is accepted or rejected.

## Universal Acceptance Gate

A run is accepted only if it proves all applicable items:

1. Correct roadmap reading order was followed.
2. Deep audit was performed before edits.
3. Files inspected are listed.
4. Current runtime/output/screenshot evidence was reviewed when artifact behavior is involved.
5. Internet research was used for unstable, technical, strategy, formula, platform, or performance claims.
6. Truth gaps are named by exact source seam, formula, function, artifact, or label.
7. Existing owner path is identified before any patch.
8. No-new-code-path declaration is included.
9. Strategy-horizon gaps are classified: scalp, intraday, restricted swing, mixed, blocked, or insufficient.
10. Regime/session/timing gaps are identified where relevant.
11. Formula/method gaps include input data, layer fit, runtime cost, false-confidence risk, and logging need.
12. Contradictions are classified: local, cross-layer, cross-artifact, horizon, regime, session, stale, unavailable, rank-risk, or wording.
13. Unavailable/garbage fields are handled: visible, hidden, dependency note, log-only, conflict flag, hydrate, or remove.
14. Speed risks are named: loops, indicator handles, CopyTicks/CopyBuffer, string bloat, file IO, HUD churn, log volume, queue stalls.
15. Repairs are bounded and match the phase.
16. Changed files are exact and preserve layout.
17. Compile risk is classified.
18. Artifact proof is inspected or honestly deferred.
19. Logging proof is inspected or honestly deferred.
20. Live proof is provided only when live output exists.
21. Remaining risks are explicit.
22. Next run recommendation is bounded.
23. Changed-files zip link is absolute last line if files changed.

## Rejection Conditions

Reject the run if any occur:

- It patches code without pre-patch seam identification.
- It creates new engines, modules, runtime lanes, artifact families, publication paths, Dossier systems, logging systems, or ranking systems without explicit operator override.
- It updates only version/admin files outside a control-only phase.
- It claims compile proof without compile evidence.
- It claims live proof without live output.
- It creates a clean strategy label from stale/unavailable/proxy-only evidence.
- It moves heavy calculations into Layer 3, Layer 4 broad loops, writers, HUD, or file IO.
- It lets Market Board, Current Focus, HUD, logging, or writers become truth owners.
- It adds indicators without handle/cache/performance strategy.
- It adds logs that can slow runtime without batching/throttling/change-only logic.
- It expands strategy labels without measurable evidence and horizon permission.
- It hides useful ambiguity by forcing a clean read.
- It removes useful artifact fields without an authorized thinning reason.
- It revives the old Doc 16/17/18 roadmap.
- It treats all 3 KB Dossiers as bugs.
- It treats file size as proof by itself.
- It pads Dossiers with unavailable garbage to increase size.
- It generates Layer 5–6 for all symbols.

## RUN066R-RUN075R Source-Only Acceptance Gate

RUN066R through RUN075R are accepted only if they remain source/control/code-edit runs and preserve the no-live rule. A run in this range fails if it asks for live output, blocks because live output is absent, claims compile/live/runtime/output proof, or uses pre-patch output to prove post-patch source.

Required proof classification for this lane:

| Proof class | Acceptance rule |
|---|---|
| live_proven | Only exact source state observed in live output may use this class. |
| source_patched_live_unproven | Any patch without fresh output must use this class or a stricter one. |
| pre_patch_live_proven_post_patch_unproven | Use when a live output exists before a patch. |
| source_supported_runtime_unobserved | Use when source exists but runtime/log/artifact output was not observed. |
| scenario_absent_untested | Use when the live scenario did not occur, such as no open positions or no pending orders. |
| output_side_signal | Use when the evidence suggests a risk but does not prove the targeted lane failed. |
| held_for_future_live | Use for proof debt intentionally deferred to RUN076R or later. |
| retired/no_longer_needed | Use only when the proof item is not required for the current boundary. |

Source-only RUN066R-RUN075R reports must include prior-report absorption, proof-debt ledger updates, a next-run source-only plan, behavior drift checks, static compile-safety checks, and `NEXT_PROMPT_SEED`.

## Dossier Folder Completion Acceptance

PHASE 2A is accepted only if future runtime/output evidence proves:

1. Closed/inactive symbols produce honest compact Layer 0–2 Dossiers.
2. Open/eligible symbols produce hydrated Layer 0–4 Dossiers.
3. Top 10/selected symbols produce Layer 0–6 deep Dossiers through existing deep/current-focus/dossier owners.
4. 3 KB open-symbol shells are eliminated or explicitly classified as `INCOMPLETE`, `STALLED`, or `DEPENDENCY_BLOCKED`.
5. 3 KB closed-symbol compact files are not treated as bugs if they contain honest Layer 0–2 truth.
6. No unavailable garbage is padded into files just to increase file size.
7. Manifest/logs show hydration class per symbol.
8. Manifest/logs show queue completed/pending/stalled counts.
9. Dossier writer does not block heartbeat or HUD interaction.
10. Layer 5–6 is not generated for all 1,200 symbols.
11. Current Focus still matches selected symbol Dossier truth.
12. Market Board does not imply deeper truth than the symbol Dossier has.
13. GlobalTop10 remains deep and correct.
14. File size is treated as diagnostic only, not proof by itself.
15. Temp → validate → promote behavior is preserved or honestly proof-held.
16. Queue/resume behavior records last completed symbol, remaining count, retry reason, and publication age.

## 3 KB Classification Standard

A future Dossier repair run must classify each observed 3 KB group as one of:

- `CLOSED_L0_L2`: honest compact closed/inactive file; not a bug.
- `OPEN_SHELL`: open/eligible symbol stayed tiny; bug/backlog/stall.
- `INFO_GARBAGE`: placeholder/unavailable padding; bug.
- `DEPENDENCY_BLOCKED`: broker/spec/quote/session data missing; log dependency.
- `BUDGET_EXHAUSTED`: write cycle ran out of time; resume later.
- `PUBLICATION_INTERRUPTED`: temp/promote or last-good issue; repair through existing IO/publication path.

## Logging Acceptance

Logging changes are accepted only if they are existing-path, structured, and lightweight.

Required proof, where relevant:

- formula input/output logs
- rank reason, penalty, conflict, freshness, and rank movement logs
- contradiction severity logs
- stale/unavailable field logs
- Dossier hydration class logs
- file-size, section-count, unavailable-count, stale-count diagnostics
- queue position, queue remaining, retry count, stall reason, write duration, and write budget logs
- Top10/deep trigger logs
- Current Focus and Market Board alignment logs
- outcome/review hooks
- change-only, batched, or throttled emission proof

## Artifact Acceptance

Dossier, Current Focus, Market Board, HUD, manifest, and logs must align or explain role differences. If one surface says clean and another says conflict, the run fails unless it creates explicit conflict display.

## Strategy Acceptance

Every strategy label must include horizon, regime fit, session/timing fit, evidence quality, primary conflict, expiry, and operator meaning.

## Ranking Acceptance

Every rank must include rank type, support reasons, penalties, conflicts, freshness/age, and why-not-higher if inside Top 10.

## Speed Acceptance

No improvement is accepted if it makes broad scanning, Dossier publication, or HUD interaction materially slower without a documented budget and proof plan.

## Final Live Truth Validation Acceptance

The final validation phase must prove compile-safe build evidence, live artifacts publish, Dossier folder hydration classes, Top 10 fills and remains deep only for entitled symbols, Current Focus updates and matches selected Dossier truth, Market Board compression aligns with Dossier truth, HUD responds quickly, ranks explain themselves, contradictions display, horizon labels are correct, regime/session/timing visible, swing rules enforced, unavailable clutter reduced, logs capture evidence without spam, queue/backlog/stall counts are visible, and runtime remains acceptable across small/medium/large symbol universes.


## Academic Strategy / AI Readiness Acceptance Standard

A future academic or AI strategy phase is accepted only if all of the following are true:

| Gate | Acceptance Requirement | Rejection Condition |
|---|---|---|
| Academic evidence | Source supports a testable method lane, not a live trading claim | Paper/guru claim is treated as proof of edge |
| Strategy method library | Candidate records evidence strength, horizon, regime, session, layer fit, runtime cost, feature requirements, false-confidence risk, logging, validation, rejection | Missing any load-bearing dimension |
| MT5 data reality | Required data exists, is cheap enough at the intended layer, and has proxy caveats marked | True order flow/volume is implied where only retail proxy data exists |
| Feature dataset spine | Feature names are stable, explainable, versionable, and loggable | Feature cannot be reconstructed or reviewed later |
| Outcome labels | Review labels include movement, MFE/MAE, expiry, invalidity, conflict quality, top-rank usefulness, false confidence, missed opportunity, and no-trade-saved-loss proxy | No later review path exists |
| Walk-forward/backtest | No lookahead, closed-bar/current-bar separation, train/test split, walk-forward, OOS, broker/session/regime review, spread/slippage assumptions | One lucky backtest is accepted |
| AI governance | Model/feature/dataset versions, training/validation windows, symbol universe, broker/server, drift, calibration, explainability, rollback, kill switch, operator approval | Black-box confidence appears on operator surface |
| Full-auto lockout | No execution API/code/path is added; AI remains advisory/paper-review only | `OrderSend`, `CTrade`, auto lot, auto SL/TP, auto close, hidden execution, or live auto management appears |
| Geometry | Invalidation, room, spread-to-room, time-to-invalidity, structure distance, horizon fit, and conflict flags are support/permission evidence only | Geometry becomes hidden entry logic |
| Performance | Broad layers remain light; heavy CopyTicks/CopyBuffer/indicator work is selected/top10/deep only until proven | Broad 60/200/1200-symbol loops become heavy |

## Required No-Execution Proof For Any Strategy / AI Run

Every future strategy/AI run must state:

- No MT5 source edited unless that run is explicitly a source patch run.
- No trading execution logic edited.
- No `OrderSend`, `OrderSendAsync`, `CTrade.Buy`, or `CTrade.Sell` added.
- No automated entry, exit, sizing, SL/TP, close, or position management added.
- No compile proof claimed unless compile output exists.
- No live proof claimed unless live output exists.

## Source Library / Strategy Card Acceptance

A future source-library, book-extraction, academic, or AI-method run is accepted only if all applicable items are true:

| Gate | Acceptance Requirement | Rejection Condition |
|---|---|---|
| Source intake | Uploaded books/files, screenshot-only titles, public research, and Aurora EA zip evidence are separated | Screenshot/title evidence is treated as if the book was fully read |
| Copyright safety | Concepts and method ideas are summarized without long copyrighted passages | Book text is dumped into roadmap files |
| Aurora reference | Current uploaded Aurora EA zip/source is inspected before implementation status claims | Worker says missing/implemented without file/card/log proof |
| Crosswalk | Every extracted concept gets a Book-to-Aurora row | Concepts float as vague ideas |
| Strategy card | Every accepted method/label has a complete card | Label survives without source, rule, MT5 data, logging, validation, and rejection criteria |
| Method ledger | Extracted/partial/missing/duplicate/rejected/future-test methods are tracked | Same weak idea gets rediscovered repeatedly |
| MT5 fit | Data availability and layer cost are explicit | True order flow/external data is implied from MT5 proxy data |
| Validation | Backtest/walk-forward/forward-review/live-output comparison/paper-review path is defined | Academic or book source becomes fake proof |
| AI corpus | Feature/outcome/rejection/operator-note records are kept advisory-only | AI confidence becomes a hidden action path |
| Full-auto lock | No execution terms are introduced except documentation-only lockout statements | Full-auto/semi-auto execution creeps into current roadmap |

## Source-Library Evidence Hierarchy

1. Aurora runtime logs/output and current source proof.
2. Aurora current uploaded EA/source zip file/card/log evidence.
3. Validated strategy cards and method ledger rows.
4. Academic research supporting a test lane.
5. Practitioner books supporting a candidate idea.
6. Screenshot-only bibliographic/title evidence.

Lower evidence may point to research. It may not override higher evidence.

Every RUN066R-RUN075R source-only run is a hard failure if either the saved report or the final operator-facing response omits a fenced `GIT HEADER SUMMARY` codeblock or a fenced `FINAL SUMMARY` codeblock.
