# ASC Scope Boundary and Admissibility Map

## Purpose

This file is the compact scope/admissibility map for active ASC work.
It states what is allowed now, what remains blocked, and what admission gate would be required to widen scope.

Tier rule for this file:
- boundary lines in the matrix are hard law
- implementation examples are required product contract only where explicitly stated
- any later optional widening remains future expansion until formally admitted

---

## Boundary law

1. If a scope lane says `not allowed now`, implementation is inadmissible even if technically possible.
2. ASC is scanner truth authority, not final signal authority.
3. SIAM / Aurora is downstream interpretation authority and may consume only lawful Layer 1–5 ASC truth in layer order.
4. Menu/settings controls may configure lawful service scope and analyst-facing projection only; they may not grant ownership jumps.
5. Deep Analysis controls are Layer 5 service controls, not scanner-signal controls.
6. Future scaffold lanes are planning placeholders only until their admission gates are explicitly passed.
7. If roadmap/office status conflicts with this file, stricter admissibility wins until resolved by explicit contract update.


## Stage 5 next-step guidance

- Stage 5 means the blueprint boundary map is consolidated; it does not mean implementation is complete.
- Future work should use this file to reject casual scope widening while implementation, schema landing, and runtime verification proceed.
- If a proposed change would make ASC own final signals, make Layer 3 directional, make Layer 4 a best-trade ranker, make Layer 5 full-universe, turn Market Board into a signal dashboard, or turn menu/settings into hidden authority, the change is non-admissible.
