# ASC Build Strategy

## Purpose

This file records the active build doctrine for Aurora Sentinel Core.
It exists to stop future passes from treating foundation as a purely in-memory stage.

---

## Core correction in this pass

### Foundation includes landed proof
Foundation is not complete when the code merely compiles, boots, and owns types.
For this rebuild, foundation must also produce visible proof that runtime truth can land correctly.

That means foundation includes:
- write path proof
- server-root publication proof
- summary scaffold
- dossier scaffold
- section homes for all five major layers
- truthful landed proof of current runtime state

### Dossier scaffold belongs early
The dossier is not a late luxury surface.
It is part of how the runtime proves itself during staged development.

### Layer richness still stays staged
Early scaffold does **not** authorize fake depth.
Each later layer must hydrate its own section honestly over time.

---

## Pre-implementation freeze phase (required)

Before MT5 rebuild implementation is admissible, ASC must pass a formal pre-implementation freeze phase with recorded evidence.

Required freeze criteria:
1. heartbeat matrix frozen
2. layer contracts frozen
3. publication matrix frozen
4. artifact lifecycle frozen
5. HUD role matrix frozen
6. symbol-action scaffold frozen

**MT5 rebuild is not admissible until all freeze criteria pass and are logged with blueprint-file evidence.**

---

## Active build order

1. Repository / office / blueprint authority
2. Pre-implementation freeze contracts and evidence
3. Compile-safe runtime surface
4. Output-proof foundation
   - write path proof
   - server-root output proof
   - summary scaffold
   - dossier scaffold for all layers
   - board/current-focus contract surfaces
5. Layer 1 market-state proof
6. Archive-truth classification work
7. Archive-inheritance HUD structural build
8. HUD tactical polish (later, bounded)
9. Menu scaffold in a later dedicated pass
10. Later layers and hardening

---

## Runtime execution model prerequisite

Before dispatcher/governor implementation or any later runtime widening begins, future passes must read and obey the runtime execution model in the active blueprint and MT5 planning surfaces.

This is not optional implementation flavor.
It is build law.

That means future passes must preserve:
- one major lane per heartbeat
- bounded sub-lane admission only
- scan -> write -> read breathing rhythm
- atomic publication windows
- cursor-based progression and bounded work budgets
- HUD as a consumer, not a hidden compute engine

Any implementation pass that invents its own execution model, mixes heavy domains casually, or treats heartbeat rhythm as negotiable is invalid even if the code compiles.

---

## What this strategy rejects

The rebuild must reject these weak interpretations:
- “foundation is done because enums exist”
- “file outputs can wait until much later”
- “the dossier can be added after the logic is rich”
- “board/current-focus contracts can wait until HUD polish”
- “HUD or menu work should jump ahead of landed truth”
- “classification can be improvised without archive evidence”

---

## Boundary reminders

### Classification
Must come later and must use archive truth.

### HUD
Must come later and must inherit from archive HUD evidence.
Do not replace it with an unrelated new design.

### Menu
Not part of this pass.
Keep it explicitly deferred.
